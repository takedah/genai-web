# オンプレミスからの名前解決設定

本ドキュメントでは、専用線/VPN 越しのオンプレミス端末から源内 Web（閉域構成）を利用するために必要な DNS（名前解決）の設定手順を説明します。

前提として、名前解決とは別に、VPC エンドポイント共通セキュリティグループと ALB のセキュリティグループがオンプレミス側 CIDR を許可している必要があります（パラメーター `closedNetworkAllowedClientCidrs`。詳細は [閉域ネットワーク対応の変更点](./閉域ネットワーク対応の変更点.md) を参照）。名前解決と経路許可の両方が揃って初めて疎通します。

## 1. クライアントが解決すべき名前

このアプリの SPA は、REST API だけでなく Web ブラウザー上の AWS SDK から一部の AWS サービスを直接呼び出します。これらのサービス名はVPC 内でだけ VPC エンドポイントのプライベート IP アドレスに解決されるため、オンプレミス端末からは何らかの方法で VPC 内のリゾルバーを経由して解決する必要があります。

| # | 解決すべき DNS 名 | 解決先 | 用途 |
|---|---|---|---|
| 1 | `cognito-idp.<リージョン>.amazonaws.com` | Cognito IDP エンドポイント | サインイン・トークン取得 |
| 2 | `cognito-identity.<リージョン>.amazonaws.com` | Cognito Identity エンドポイント | 一時クレデンシャル取得 |
| 3 | `*.execute-api.<リージョン>.amazonaws.com` | API Gateway エンドポイント | REST API（メイン + チーム管理） |
| 4 | `lambda.<リージョン>.amazonaws.com` | Lambda エンドポイント | チャットストリーミング応答の直接 Invoke |
| 5 | `s3.<リージョン>.amazonaws.com` と `*.s3.<リージョン>.amazonaws.com` | S3 Interface エンドポイント | 署名付き URL でのファイル送受信（バケット名付きの仮想ホスト形式で解決されるためワイルドカードが必要） |
| 6 | `closedNetworkDomainName` に設定したドメイン | 内部 ALB | アプリ本体の画面 |

1〜5 以外の VPC エンドポイント（Bedrock、Transcribe など）は VPC 内の Lambda / Fargate からしか呼ばれないため、クライアント側の名前解決は不要です。

## 2. 基本パターン: アプリ VPC に Route 53 インバウンドエンドポイントを用意する場合

アプリ（源内 Web）の VPC に Route 53 Resolver インバウンドエンドポイントを作成し、オンプレミスの DNS サーバーから上記 6 ゾーンを条件付きフォワードします。

## 3. 共有リソース VPC のインバウンドエンドポイントを経由する場合

オンプレミスのリゾルバー DNS が、既に別の VPC（いわゆる共有リソース VPC）の Route 53 インバウンドエンドポイントと連携しており、`amazonaws.com` も既に条件付きフォワードの対象になっているケースの手順です。

### 3-1. なぜそのままでは解決できないか

共有リソース VPC にセクション 1 の 1〜5 に対応するインターフェース型 VPC エンドポイント（Private DNS 有効）が既にある場合、共有リソース VPC の Route 53 インバウンドエンドポイントで必要な名前解決が既にできる状態なので、以下の手順は不要です。

そうでない場合、共有リソース VPC の Route 53 インバウンドエンドポイントで名前解決を行うと、`cognito-idp.…` などがパブリック IP アドレスに解決されてしまい、閉域からは到達できません。

そこで、セクション 1 の 1〜5 に対応する Route 53 Private Hosted Zone を手動で作成し、アプリ VPC のエンドポイントへの Alias レコードを設定したうえで、共有リソース VPC に関連付けます。アプリドメイン（セクション 1 の 6）は、スタックが作成した Route 53 Private Hosted Zone を共有リソース VPC に追加で関連付けます。

### 3-2. 事前確認（重要）・共有リソース VPC に同じサービスのエンドポイントがないか

**共有リソース VPC に同じサービスの VPC エンドポイント（Private DNS 有効）が既にある場合、そのサービスの Route 53 Private Hosted Zone を作成・関連付けしてはいけません。** 共有 VPC のリゾルバーを使う全システム（オンプレミス全体・共有 VPC 内のワークロード・リゾルバールール経由の他 VPC）の解決結果がアプリ VPC のインターフェース VPC エンドポイントの IP アドレスに変わり、**他システムの通信障害の原因になります**。

まず共有リソース VPC 側のエンドポイントを確認してください（共有 VPC のアカウントで実行）:

```bash
aws ec2 describe-vpc-endpoints \
  --filters "Name=vpc-id,Values=<共有リソース VPC の ID>" \
  --query 'VpcEndpoints[].{Service:ServiceName,PrivateDns:PrivateDnsEnabled}'
```

サービスごとに以下のとおり状況を確認します（状況が混在して構いません）。

| 共有リソース VPC の状況 | 取るべき方法 |
|---|---|
| そのサービスのエンドポイントが既にある | Route 53 Private Hosted Zone は作らず、共有リソース VPC の既存 VPC エンドポイントをそのまま利用する。既存エンドポイントの SG がオンプレミス CIDR を許可しているか、エンドポイントポリシーが制限されていないかを確認 |
| そのサービスのエンドポイントがない | 本ドキュメントの手順（3-3 以降）で Route 53 Private Hosted Zone を作成 |
| アプリドメイン（`closedNetworkDomainName`） | 固有のドメイン名なので衝突せず、関連付け（3-4）は常に安全 |

**例外として execute-api（API Gateway）だけは注意が必要です。** 本プロジェクトの REST API はリソースポリシーで `aws:SourceVpce` = アプリ VPC の API Gateway エンドポイント ID のみを許可しているため、共有リソース VPC の既存 execute-api エンドポイント経由のリクエストは失敗します。共有リソース VPC に execute-api の VPC エンドポイントが既にある場合は、リソースポリシーの許可対象に共有リソース VPC 側のエンドポイント ID を追加するよう CDK のパラメーター追加が必要です。

### 3-3. 手順（1〜5 のゾーン作成）

以下、リージョンは `ap-northeast-1`、Cognito IDP を例にします。他の 4 サービスも同じ手順の繰り返しです。

**(1) デプロイ後、アプリ VPC の VPC エンドポイントの DNS エントリを確認します。**

```bash
aws ec2 describe-vpc-endpoints \
  --filters "Name=service-name,Values=com.amazonaws.ap-northeast-1.cognito-idp" \
  --query 'VpcEndpoints[0].DnsEntries[0]'
```

出力例（`DnsName` と `HostedZoneId` を手順 (3) で使います）:

```json
{
  "DnsName": "vpce-0123456789abcdef0-abcdefgh.cognito-idp.ap-northeast-1.vpce.amazonaws.com",
  "HostedZoneId": "Z2E726K9Y6RL4W"
}
```

**(2) サービス名と同名の Private Hosted Zone を、共有リソース VPC に関連付けて作成します。**

```bash
aws route53 create-hosted-zone \
  --name cognito-idp.ap-northeast-1.amazonaws.com \
  --vpc VPCRegion=ap-northeast-1,VPCId=<共有リソース VPC の ID> \
  --caller-reference genai-web-$(date +%s) \
  --hosted-zone-config Comment="genai-web closed network",PrivateZone=true
```

**(3) VPC エンドポイントへの Alias レコード（A）を作成します。**

```bash
aws route53 change-resource-record-sets \
  --hosted-zone-id <手順 (2) で作成したゾーンの ID> \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "cognito-idp.ap-northeast-1.amazonaws.com",
        "Type": "A",
        "AliasTarget": {
          "HostedZoneId": "<手順 (1) の HostedZoneId>",
          "DNSName": "<手順 (1) の DnsName>",
          "EvaluateTargetHealth": false
        }
      }
    }]
  }'
```

**(4) サービスごとのゾーン名とレコード名は以下のとおりです。**

| ゾーン名 | 作成するレコード名 |
|---|---|
| `cognito-idp.ap-northeast-1.amazonaws.com` | ゾーン頂点（ゾーン名と同じ） |
| `cognito-identity.ap-northeast-1.amazonaws.com` | ゾーン頂点 |
| `execute-api.ap-northeast-1.amazonaws.com` | `*.execute-api.ap-northeast-1.amazonaws.com`（ワイルドカード） |
| `lambda.ap-northeast-1.amazonaws.com` | ゾーン頂点 |
| `s3.ap-northeast-1.amazonaws.com` | ゾーン頂点 と `*.s3.ap-northeast-1.amazonaws.com` の 2 レコード |

### 3-4. 手順（アプリドメインのゾーン関連付け）

スタックが作成した Private Hosted Zone（`closedNetworkDomainName` のゾーン）を共有リソース VPC に追加で関連付けます。

```bash
# ゾーン ID の確認
aws route53 list-hosted-zones-by-name --dns-name <closedNetworkDomainName> \
  --query 'HostedZones[0].Id'

# 共有リソース VPC への関連付け
aws route53 associate-vpc-with-hosted-zone \
  --hosted-zone-id <上記のゾーン ID> \
  --vpc VPCRegion=ap-northeast-1,VPCId=<共有リソース VPC の ID>
```

### 3-5. 共有リソース VPC が別 AWS アカウントの場合

ゾーンの作成・関連付けに承認の手順が加わります（AWS マネジメントコンソールでは操作できず、CLI / API のみ）。

```bash
# ゾーンを所有するアカウント側で、関連付けを承認
aws route53 create-vpc-association-authorization \
  --hosted-zone-id <ゾーン ID> \
  --vpc VPCRegion=ap-northeast-1,VPCId=<共有リソース VPC の ID>

# 共有リソース VPC を所有するアカウント側で、関連付けを実行
aws route53 associate-vpc-with-hosted-zone \
  --hosted-zone-id <ゾーン ID> \
  --vpc VPCRegion=ap-northeast-1,VPCId=<共有リソース VPC の ID>
```

3-3 のように「作成時に共有リソース VPC を関連付ける」ことはできないため、いったんアプリ VPC などの自アカウントの VPC に関連付けて作成し、上記の承認 → 関連付け → （必要なら）元 VPC の関連付け解除、の順に実施してください。

## 4. 運用上の注意

- **これらのゾーン・レコードは CDK 管理外です。** スタックの作り直し等で VPC エンドポイントが再作成されると Alias のターゲット（`DnsName` / `HostedZoneId`）が変わるため、手順 (1), (3) を再実施して更新してください。
- 新しい AWS サービスを Web ブラウザーから直接呼ぶ機能を追加した場合は、VPC エンドポイントの追加（`closed-vpc.ts`）とあわせて、本手順のゾーン追加も必要です。
