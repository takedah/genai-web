# 源内 Web の閉域ネットワーク対応の変更点

本ドキュメントは、デジタル庁 [genai-web（源内 Web）](https://github.com/digital-go-jp/genai-web) を元に、本プロジェクトがプライベートサブネットのみの閉域ネットワーク環境で動作するように行った変更内容をまとめたものです。

## 1. 全体像

元の源内 Web はインターネット公開を前提としたアーキテクチャでしたが、本プロジェクトでは、全ての通信を VPC 内の閉域ネットワークで完結させる構成に変更しています。源内 Web が元にしている [Generative AI Use Cases (GenU)](https://github.com/aws-samples/generative-ai-use-cases) の閉域モードのコードを移植したのがベースです。

| 源内 Web | 本プロジェクト（閉域） |
|---|---|
| CloudFront + WAF | 内部 ALB + Fargate（Express/S3 配信） |
| API Gateway（Regional/公開） | API Gateway（Private + リソースポリシー） |
| Lambda（VPC外） | Lambda（VPC内） |
| AWS API へ直接（インターネット） | VPC エンドポイント経由 |
| 公開 ACM ドメイン / Route53 公開ゾーン | ACM（同一リージョン） / Route53 Private Hosted Zone |
| SAML SSO / Cognito Hosted UI | 削除（閉域で動作不可） |
| WAF による IP アドレス制限 | VPC 到達性による制御 |

## 2. VPC とネットワーク基盤の新設

### `packages/cdk/lib/construct/closedNetwork/closed-vpc.ts`（新規）

閉域環境の心臓部です。

- `PRIVATE_ISOLATED` サブネットのみ。NAT ゲートウェイもインターネットゲートウェイも持たず、外部への経路が一切ありません。
- AWS サービスへはインターフェース型 VPC エンドポイントとゲートウェイ型 VPC エンドポイントで到達します（詳細は「3. 閉域化に必要な VPC エンドポイントの整備」）。
- 全エンドポイントで `privateDnsEnabled: true` を設定し、通常のサービスエンドポイント名が VPC 内で自動的にプライベート IP アドレスへ名前解決されるようにしています。
- Route53 Private Hosted Zone を作成し、閉域内でのカスタムドメイン名前解決に対応。`closedNetworkPrivateHostedZoneId` を指定した場合は新規作成せず既存ゾーンを取り込みます。
- VPC フローログ（全トラフィック、CloudWatch Logs へ 3 ヶ月保持）を有効化。閉域では SG やエンドポイント設定漏れによる疎通不良の調査が難しいため。

### `packages/cdk/lib/closed-network-stack.ts`（新規）

この VPC 構成を独立したスタックとして切り出しています。閉域モードではクロスリージョン参照ができないため、`region === modelRegion`（アプリのリージョンとモデルのリージョンの一致）を強制チェックしています。

## 3. 閉域化に必要な VPC エンドポイントの整備

### 通信経路ごとの洗い出し

#### 利用者のブラウザから直接呼ばれるサービス

このアプリの SPA は、API Gateway 経由の REST API だけでなく、Web ブラウザー上の AWS SDK から一部の AWS サービスを直接呼び出す構造になっています。閉域では利用者端末も（専用線や VPN 等で）VPC に閉じたネットワークからアクセスするため、これらの呼び出し先全てに VPC エンドポイントが必要です。

| サービス | 呼出し元の処理 |
|---|---|
| Cognito IDP | サインイン・トークン取得（SDK 経由の認証 API） |
| Cognito Identity | Identity Pool からの一時クレデンシャル取得 |
| API Gateway | REST API 全般の受け口。Private API 化の前提となるエンドポイントで、リソースポリシーの `aws:SourceVpce` にもこの ID を使用 |
| Lambda | `predictStream` の直接 Invoke（チャットのストリーミング応答は Web ブラウザーが Lambda を直接呼ぶ実装のため） |
| S3（Interface） | 署名付き URL によるファイル添付・文字起こし音声のアップロード / ダウンロード |

#### VPC 内に移した Lambda / Fargate が呼ぶサービス

Lambda を VPC 内に移したため、Lambda が行っていた AWS サービス呼出しも全て VPC エンドポイント経由に切り替わります。Fargate（SPA 配信）の分も含みます。

| サービス | 呼出し元の処理 |
|---|---|
| Bedrock Runtime | チャット・画像生成などのモデル推論 |
| Transcribe | 音声文字起こしジョブの開始・結果取得（`startTranscription` / `getTranscription` Lambda から呼び出し） |
| S3（Gateway + Interface） | ファイルアップロード・SPA アセット配信元・ログ保管。VPC 内からの通常アクセスは Gateway 型、ブラウザからの署名付き URL アクセス向けに Interface 型も併設 |
| DynamoDB（Gateway） | チャット履歴・チーム管理データの読書き |
| CloudWatch Logs | Lambda / Fargate のログ出力 |
| STS | AssumeRole・一時クレデンシャル発行 |
| ECR / ECR Docker | Fargate タスク起動時のコンテナイメージ取得 |

#### 源内 Web 独自機能（チーム管理・exApp）が呼ぶサービス

GenU には無く源内 Web で追加された機能のための VPC エンドポイントです。

| サービス | 呼出し元の処理 |
|---|---|
| Secrets Manager | exApp（外部マイクロサービス）呼び出し用の認証情報取得 |
| SQS | exApp のステータスポーリングキュー |
| KMS | ユーザー ID 生成の HMAC（`GenerateMac`）など Lambda からの暗号化キー操作 |
| SES | Custom Email Sender によるサインアップ・MFA 等のメール送信 |

### 名前解決とセキュリティグループ

- 全ての Interface 型エンドポイントで `privateDnsEnabled: true` を設定。アプリは通常のサービスエンドポイント名を使うだけで VPC 内のプライベート IP アドレスに解決されるため、アプリケーションコードの変更なしに呼び出し先が閉域経路に切り替わります。
- Interface 型エンドポイントは 1 つのセキュリティグループを共有し、VPC CIDR と `closedNetworkAllowedClientCidrs`（利用者端末側の CIDR）からの 443/TCP のみ許可しています。ブラウザは Cognito・API Gateway・Lambda・S3（署名付き URL）をエンドポイント経由で直接呼び出すため、専用線/VPN 越しのオンプレミス端末から利用する場合は `closedNetworkAllowedClientCidrs` の設定が必須です（未設定だとサインインもできません）。

### 運用上の注意

新しい AWS サービスを使う機能を追加した場合は、対応する VPC エンドポイントを `closed-vpc.ts` の `VPC_ENDPOINTS` に追加する必要があります（`closed-vpc.test.ts` のエンドポイント数の期待値もあわせて更新してください）。

なお、コード監査の結果、SDK 呼び出し・IAM 権限とも存在しないことが確認できた Polly（音声合成）と EventBridge（`PutEvents`）のエンドポイントは削除済みです（Interface 型エンドポイントは 1 つあたり月額約 $20 × 2 AZ の固定費がかかるため）。将来これらのサービスを使う機能を実装する場合は再追加してください。ログ収集のスケジュール起動（EventBridge → Lambda）は AWS 側からの呼び出しのためエンドポイント不要です。

なお、Bedrock Agents（Classic）は 2026 年 7 月 30 日以降新規利用不可となったため、エージェント機能（`bedrock-agent` / `bedrock-agent-runtime` への依存コード・IAM 権限・VPC エンドポイント）は削除済みです。将来エージェント機能が必要になった場合は、後継サービスの Bedrock AgentCore（PrivateLink 対応済み）で実装し、`bedrock-agentcore` の VPC エンドポイントを `VPC_ENDPOINTS` に追加してください。

## 4. フロントエンド配信の変更（CloudFront → ALB + ECS）

最も大きなアーキテクチャ変更です。CloudFront は公開エッジサービスであり閉域では使えないため、内部 ALB + Fargate コンテナが S3 から静的ファイルを配信する方式に置き換えています。

### `packages/cdk/lib/construct/closedNetwork/closed-web.ts`（新規）

内部向け ALB
: `internetFacing: false`、`PRIVATE_ISOLATED` サブネット。セキュリティグループはリスナーポートを VPC CIDR + `closedNetworkAllowedClientCidrs` からのみ許可（VPC エンドポイントの SG と同じ方針）。アクセスログを専用 S3 バケット（SSE-S3、90 日で失効）に出力。

Fargate サービス
: `ApplicationLoadBalancedFargateService` で SPA 配信コンテナを起動。2 AZ を活かすため常時 2 タスク以上とし、CPU 使用率 50% で 2〜20 台にオートスケール。デプロイサーキットブレーカー（自動ロールバック）を有効化。

SPA アセット用の S3 バケット
: スタック共通の CMEK（KMS）で暗号化し、コンテナのタスクロールに読み取り権限を付与。

監視
: `Monitoring` construct に ALB 5xx（ELB 生成 / ターゲット生成）・ヘルスチェック失敗・ECS CPU 高止まりのアラームを追加し、既存の SNS トピック（Slack 通知）に接続。

### `packages/cdk/fargate-s3-server/`（新規）

- CloudFront の代替となる軽量な Express サーバー（`app.ts`）で、リクエストされたキーを S3 から取得して返します。拡張子のないパス（SPA のルーティング。例: `/chat/xxx`）は `index.html` にフォールバックし、存在しないアセット（拡張子付きパス）には 404 を返します。`/healthcheck` は ALB のヘルスチェック用。
- CloudFront が担っていた以下の責務もこのサーバーが引き継ぎます。
  - セキュリティヘッダの付与（CSP / HSTS / X-Frame-Options / X-Content-Type-Options / Referrer-Policy。上流の ResponseHeadersPolicy と同一内容）
  - キャッシュ制御（`index.html` は `no-store`、ハッシュ付きアセット `assets/` は `max-age=31536000, immutable`）
  - gzip 圧縮、ETag の転送と `If-None-Match` による 304 応答
- Dockerfile でコンテナ化し ECR にプッシュ。（ビルドは `LINUX_AMD64`）

### `packages/cdk/lib/construct/web.ts`（変更）

- CloudFront / ResponseHeadersPolicy / セキュリティヘッダ / CSP / メンテナンス用 CloudFront Function など、CloudFront 関連のコードを全て削除。
- フロントエンドのビルド（`NodejsBuild`）は残しつつ、出力先を `ClosedWeb` が作った `webBucket` に直接デプロイするよう変更。

## 5. API Gateway の Private 化

`api.ts` と `team-access-control.ts`（源内 Web が GenU に追加したチーム管理機能）の両方で、REST API を Private エンドポイント化しています。

- `EndpointType.PRIVATE` + API Gateway 用 VPC エンドポイントを指定。
- リソースポリシーで、指定した VPC エンドポイント（`aws:SourceVpce`）経由のアクセスのみを許可。これにより API がインターネットから到達不能になります。
- チーム管理 API も `REGIONAL` から `PRIVATE` へ変更。cdk-nag（セキュリティチェックツール）の警告抑制理由も「WAF で保護」から「VPC エンドポイント経由のみアクセス可能」に更新。

## 6. Lambda の VPC 内デプロイ

元は Lambda は VPC 外（インターネット経由で AWS API を叩く）でした。閉域では VPC 内に置き、VPC エンドポイント経由で通信させます。

- `api.ts` / `auth.ts` / `transcribe.ts` / `team-access-control.ts` の全ての Lambda に、共通の VPC 設定 `lambdaVpcProps`（`vpc` + `PRIVATE_ISOLATED` サブネット）を追加。
- exApp 呼び出し用に個別に作っていた VPC（`invoke-exapp-lambda-vpc.ts`）は不要になったため削除し、共通の閉域 VPC に統合。パラメーター `vpcIdForInvokeExApp` も廃止。

## 7. 認証（SAML / セルフサインアップ / IP アドレス制限）の削除

閉域では Cognito Hosted UI が VPC エンドポイントで動作しないため、それに依存する機能を削除しています。

### `packages/cdk/lib/construct/auth.ts`（変更）

SAML 認証を全面削除
: `samlAuthEnabled` / OAuth 設定（コールバック URL、IdP プロバイダ）/ UserPoolClient の OAuth フローをすべて除去。

WAF ベースの IP アドレス制限を削除
: `allowedIpV4/V6AddressRanges` による `DENY` ポリシー（`aws:SourceIp` 条件）を削除。

### `generative-ai-use-cases-stack.ts`

SAML 用の IdP 名リスト構築、コールバック URL 設定、CloudFront ドメインを使った UserPoolClient のオーバーライド、SAML 関連の CfnOutput を全て削除。

この改変により、`docs/カスタムドメイン設定.md` と `docs/SAML認証手順.md` の機能は使えなくなっています。認証は Cognito のユーザー名 / パスワード（+ 任意で Email MFA）に限定され、ロール制御は Cognito グループベースになります。

フロントエンド側の SAML 分岐（`main.tsx` の `AuthWithSAML` への切り替えと `VITE_APP_SAML*` 環境変数）も削除済みで、常にユーザープール認証（`AuthWithUserpool`）を使用します。

## 8. Transcribe ストリーミング API について

源内 Web のコード内で Transcribe ストリーミング API は使われていないため、不要な権限付与のコードを削除しています。

- `auth.ts`：3 つのロール（SystemAdmin / TeamAdmin / User）から `transcribe:StartStreamTranscriptionWebSocket` を削除
- `transcribe.ts`：`authenticatedRole` への同権限の付与（`GrantAccessTranscribeStream`）を削除

## 9. 削除されたスタック・コンストラクト

| ファイル | 役割 | 削除理由 |
|---|---|---|
| `cloud-front-waf-stack.ts` | CloudFront 用 WAF（us-east-1） | CloudFront 廃止に伴い不要 |
| `common-web-acl.ts` | WAF ACL 定義 | 同上 |
| `app-domain-stack.ts` | 公開カスタムドメイン / ACM 証明書 | 公開ドメイン不要 |
| `invoke-exapp-lambda-vpc.ts` | exApp 呼び出し用の個別 VPC | 共通閉域 VPC に統合 |

`create-stacks.ts` ではこれらのスタック生成を削除し、代わりに `ClosedNetworkStack` を生成。メインスタックに VPC・API Gateway VPC エンドポイント・PrivateHostedZone を渡し、依存関係（`addDependency`）を設定しています。

## 10. パラメーター（設定）の変更

デプロイ時の設定パラメーターは `packages/cdk/lib/stack-input.ts` でスキーマ（型と入力チェック）が定義されており、ここが以下のように変わっています。

### 削除されたパラメーター

- SAML 認証一式（`samlAuthEnabled` など）
- WAF 系（`allowedIpV4/V6AddressRanges`、`allowedCountryCodes`）
- 公開カスタムドメイン系（`useHostedZone` / `hostName` / `domainName` / `hostedZoneId` / `certificateArn`）
- `vpcIdForInvokeExApp`
- SAML 関連の入力チェック（zod スキーマの `refine`）

### 新規追加されたパラメーター

| パラメーター | 説明 |
|---|---|
| `closedNetworkVpcCidr` | 閉域 VPC の CIDR（デフォルト `10.1.0.0/16`） |
| `closedNetworkDomainName` | ドメイン名（必須） |
| `closedNetworkCertificateArn` | ACM 証明書 ARN（必須、アプリと同一リージョンであること。us-east-1 ではない点に注意） |
| `closedNetworkPrivateHostedZoneId` | 既存プライベートホストゾーン ID（任意）。指定した場合は新規ゾーンを作成せず、既存ゾーンに ALB への A レコードを作成する。ゾーン名は `closedNetworkDomainName` と一致している必要があり、利用者がアクセスしてくる VPC にそのゾーンが関連付け済みであることが前提 |
| `closedNetworkAllowedClientCidrs` | 利用者端末（オンプレミス）側の CIDR リスト（デフォルト `[]`）。VPC エンドポイント共通 SG と ALB の SG の許可対象に追加される。専用線/VPN 越しのオンプレミス端末から利用する場合は必須 |

## 11. CDK コードの変更点詳細

セクション 2〜10 の内容を、CDK コードの観点から具体的に補足します。

### 11.1 スタック構成の変更（`create-stacks.ts`）

スタックの構成そのものが変わっています。

**元の源内 Web（4 スタック構成）**

| スタック | リージョン | 役割 |
|---|---|---|
| `AppDomainStack` | us-east-1 | 公開カスタムドメインの ACM 証明書・ホストゾーン |
| `CloudFrontWafStack` | us-east-1 | CloudFront 用 WAF（条件付きで生成） |
| `GuardrailStack` | modelRegion | Bedrock Guardrail（条件付きで生成） |
| `GenerativeAiUseCasesStack` | アプリのリージョン | メインスタック |

CloudFront 用の WAF と ACM 証明書は us-east-1 にしか置けないため、スタックを分割し `crossRegionReferences: true` で us-east-1 のリソース（証明書 ARN、WAF ACL ARN）をメインスタックへクロスリージョン参照していました。

**本プロジェクト（3 スタック構成）**

| スタック | リージョン | 役割 |
|---|---|---|
| `ClosedNetworkStack` | アプリのリージョン | 閉域 VPC・VPC エンドポイント・PrivateHostedZone |
| `GuardrailStack` | modelRegion | 変更なし |
| `GenerativeAiUseCasesStack` | アプリのリージョン | メインスタック |

- `AppDomainStack` と `CloudFrontWafStack` を削除し、`ClosedNetworkStack` を新設。
- us-east-1 依存が消えたため、全てのスタックが同一リージョンに収まります（`ClosedNetworkStack` は `region: params.region` で生成）。クロスリージョン参照が発生しなくなったため、`crossRegionReferences: true` の指定も削除しています。
- `ClosedNetworkStack` の出力（`vpc` / `apiGatewayVpcEndpoint` / `hostedZone`）をメインスタックの props として受け渡し、`generativeAiUseCasesStack.addDependency(closedNetworkStack)` で明示的な依存関係を設定。

さらに `closed-network-stack.ts` では、閉域モードではクロスリージョン参照ができないことから、デプロイ時に以下のガードを入れています。

```typescript
if (this.region !== modelRegion) {
  throw new Error(
    `The app region and modelRegion must match in closed network mode ...`,
  );
}
```

### 11.2 メインスタックへの設定値（props）の受け渡し（`generative-ai-use-cases-stack.ts`)

メインスタックが外から受け取る設定値の型定義（`GenerativeAiUseCasesStackProps`）が以下のように変わりました。

```typescript
// 削除された props
webAclId?: string;        // WAF (CloudFront)
cert?: ICertificate;      // us-east-1 の ACM 証明書
hostedZoneId?: string;    // 公開ホストゾーン

// 追加された props
vpc: ec2.IVpc;                                    // 閉域 VPC
apiGatewayVpcEndpoint: ec2.InterfaceVpcEndpoint;  // Private API 用エンドポイント
hostedZone?: IHostedZone;                         // プライベートホストゾーン（新規作成または既存取り込み）
```

この `vpc` と `apiGatewayVpcEndpoint` が、スタック内の各コンストラクト（`Auth` / `Api` / `Transcribe` / `TeamAccessControlStack` / `LoggingStack` / `ClosedWeb`）へ一律に配られる構造です。

また、デプロイ完了時に表示されるアプリの URL（`WebUrl` という CfnOutput）も、CloudFront ドメインではなく PrivateHostedZone のゾーン名（無ければ ALB の DNS 名）を出力するように変更されています。

### 11.3 Lambda の VPC 配置パターン（全コンストラクト共通）

各コンストラクトの先頭で共通の VPC 設定オブジェクトを 1 つ定義し、全ての Lambda 関数定義（`NodejsFunction`）に TypeScript のスプレッド構文（`...変数名`。オブジェクトの中身をその場に展開する書き方）で差し込むパターンで統一されています。

```typescript
const lambdaVpcProps: Pick<NodejsFunctionProps, 'vpc' | 'vpcSubnets'> = {
  vpc: props.vpc,
  vpcSubnets: { subnetType: ec2.SubnetType.PRIVATE_ISOLATED },
};

new NodejsFunction(this, 'PredictStream', {
  runtime: Runtime.NODEJS_22_X,
  entry: './lambda/predictStream.ts',
  ...lambdaVpcProps,  // ← 全 Lambda にこれを追加
  // ...
});
```

適用範囲は以下のとおりで、アプリ内の全ての Lambda が対象です。

| コンストラクト | 対象 Lambda |
|---|---|
| `api.ts` | predict / predictStream / predictTitle / generateImage / チャット・メッセージ・システムコンテキスト CRUD / 署名付き URL 発行 / ファイル削除 など約 20 関数 |
| `auth.ts` | checkEmailDomain / assignUserToDefaultGroupOnSignUp / customEmailSender |
| `transcribe.ts` | getFileUploadSignedUrl / startTranscription / getTranscription |
| `team-access-control.ts` | チーム管理 API 群・getArtifactFile / invokeExApp / pollExAppStatus |
| `logging.ts` | Cognito ログエクスポート / DynamoDB 日次エクスポート / ログ転送系の全 4 関数 |

`logging.ts` / `logging-stack.ts` にも `vpc` を受け取る props が追加されており、ログ収集系の Lambda も漏れなく VPC 内に配置されています。

なお、`team-access-control.ts` では元コードで exApp 呼び出し Lambda だけが専用 VPC（既存 VPC を `vpcId` で指定するか、`InvokeExAppLambdaVpc` を新規作成）に置かれていましたが、この分岐ロジックを丸ごと削除し、共通の `lambdaVpcProps` に統一しています。これに伴い、`team-access-control-stack.ts`（メインスタックの中に入れ子で作られる NestedStack）が受け取る設定値も、VPC の ID 文字列（`vpcId: string | undefined`）から VPC オブジェクトそのもの（`vpc: ec2.IVpc`）+ `apiGatewayVpcEndpoint` へ変更されています。

### 11.4 API Gateway の Private 化の実装

`api.ts` と `team-access-control.ts` の両方の `RestApi` に、同じパターンで以下が追加されています。

```typescript
const api = new RestApi(this, 'Api', {
  endpointConfiguration: {
    types: [EndpointType.PRIVATE],
    vpcEndpoints: [props.apiGatewayVpcEndpoint],
  },
  policy: new PolicyDocument({
    statements: [
      new PolicyStatement({
        effect: Effect.ALLOW,
        principals: [new AnyPrincipal()],
        actions: ['execute-api:Invoke'],
        resources: ['execute-api:/*'],
        conditions: {
          StringEquals: {
            'aws:SourceVpce': props.apiGatewayVpcEndpoint.vpcEndpointId,
          },
        },
      }),
    ],
  }),
  // ...
});
```

ポイントは 2 段構えになっていることです。

1. `EndpointType.PRIVATE` + `vpcEndpoints`: API を Private エンドポイントとして作成し、指定 VPC エンドポイントに Route53 エイリアスを関連付け
2. リソースポリシー: 一旦 `AnyPrincipal`（全ての呼び出し元）を許可したうえで、`aws:SourceVpce` 条件によりこの VPC エンドポイント経由のリクエストだけに絞込み

なお、元コードで WAF を条件付きで REST API・UserPool に関連付けていた `CfnWebACLAssociation`（3 箇所）は削除されています。

### 11.5 フロントエンド配信の実装（`ClosedWeb` + `fargate-s3-server`）

`closed-web.ts` は `ApplicationLoadBalancedFargateService`（ALB・ECS クラスター・Fargate サービス・ターゲットグループなどを一括作成する L3 コンストラクト）を使い、以下を構築します。

- 内部 ALB（`internetFacing: false`、`PRIVATE_ISOLATED` サブネット、SG はリスナーポートを VPC CIDR + `closedNetworkAllowedClientCidrs` のみ許可、アクセスログを S3 に出力）
- Fargate サービス（cpu 256 / memory 512MiB / desiredCount 2、CPU 使用率 50% で 2〜20 タスクにオートスケール、デプロイサーキットブレーカーによる自動ロールバック付き）
- コンテナイメージは `ContainerImage.fromAsset('./fargate-s3-server')` により `cdk deploy` 時にローカルでビルドされ、CDK が管理する ECR に push されます（別途イメージレジストリの準備は不要）
- `certificateArn` と `hostedZone` が揃っている場合のみ HTTPS リスナー + ALB への A レコード（エイリアス）を作成

`fargate-s3-server` の Dockerfile はマルチステージビルドで、ベースイメージに `public.ecr.aws`（Amazon ECR Public）の `node:22-slim` を使用しています（再現性のためバージョン固定。実行ステージは非 root ユーザー・`NODE_ENV=production`）。ECS のタスク停止時は SIGTERM を受けて処理中のリクエスト完了を待ってから終了します。

`web.ts` 側は、フロントエンドのビルドに元コードと同じ `NodejsBuild`（`@cdklabs/deploy-time-build`）を使い続けています。これはデプロイ時に CodeBuild 上で `npm run web:build` を実行して成果物を S3 に配置する仕組みで、変更点は配置先だけです。

- 元：`CloudFrontToS3` コンストラクトが作る CloudFront オリジンバケット
- 本プロジェクト：`ClosedWeb` が作る `webBucket`（SPA の配信元）

CloudFront 固有だったコードは全て削除しています（ResponseHeadersPolicy、メンテナンスモード用 CloudFront Function、KMS 暗号化バケット群、`distribution` の公開プロパティ）。ただしセキュリティヘッダ/CSP とキャッシュ制御は機能としては失わず、`fargate-s3-server` が同一内容を付与する形で移植しています。

メンテナンスモードは Fargate 配信サーバーの `MAINTENANCE` 環境変数で制御する方式に変わっています。`maintenance: true` でデプロイすると ECS タスク定義の環境変数が更新され、配信サーバーは `/healthcheck` を除く全リクエストに対して S3 上の `maintenance.html`（フロントエンドのデプロイ時にバケットへコピーされる）を HTTP 503 で返します。切り替えはパラメーター変更 + `cdk deploy`（ECS タスクの入れ替えのみ）で完結し、フロントエンドの再ビルドは不要です。
